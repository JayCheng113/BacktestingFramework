## Part 1 — V2.19.0 Round 3 Polish

### S1: Strengthened P2 #2 regression test with price-dependent canary

**File:** `tests/test_guards/test_round2_regressions.py`

The original `_MutatingPortfolio` returned a constant equal-weight `{s: 1.0/n for s in panel}`. Mutating the panel had **zero effect** on the output, so the test passed even WITHOUT the round-2 fresh-panel fix.

Replaced with `_PriceDependentMutatingPortfolio`:

```python
class _PriceDependentMutatingPortfolio:
    warmup_period = 0
    def generate_weights(self, panel, target_date, prev_w, prev_r):
        sym = next(iter(panel))
        price = float(panel[sym]["adj_close"].iloc[-1])
        panel[sym].loc[:, "adj_close"] = 0.0  # in-place poison
        if price > 1.0:
            n = len(panel)
            return {s: 1.0 / n for s in panel}  # sum = 1.0
        return {s: 100.0 for s in panel}  # sum = 500, blocks
```

**Verified empirically**: under shared-panel infra (the bug), 4/5 dates produce sum=500 → BLOCK. Under fresh-panel infra (the fix), all 5 dates produce sum=1.0 → PASS. **Strong canary** — distinguishes the two cases.

Same pattern applied for `NonNegativeWeightsGuard` in a new test `test_non_negative_weights_guard_uses_fresh_panel_per_date`.

### S2: P1 #1 test cleanup leaked `tests.test_guards.*` registry entries

**File:** `tests/test_guards/test_round2_regressions.py`

Test classes defined inside the function got auto-registered into `Factor._registry` and `Factor._registry_by_key` via `__init_subclass__`, BEFORE the test renamed `__module__`. The forward-scan cleanup popped only the post-rename keys, leaving the original `tests.test_guards.*` entries as zombies.

Fix: snapshot `Factor._registry` and `Factor._registry_by_key` BEFORE creating any test classes; in `finally`, hard-restore from the snapshot via `clear() + update()`. Verified zero leaked keys after test completion.

### S3: `_unique_probe_module_name` docstring corrected

**File:** `ez/testing/guards/suite.py`

The original docstring claimed the unique name "sidesteps the registry altogether — the probe class lands in a different dict entry, avoids the collision warning." Both wrong:
- The name-keyed `_registry` (for Factor / CrossSectionalFactor / PortfolioStrategy) STILL displaces the production class transiently
- The collision warning DOES emit (until S7 fix below)

Updated docstring to explain what unique-name actually achieves (full-key conflict avoidance), what it doesn't (name-key displacement), and how `drop_probe_module` repairs the displacement via reverse walk.

### S4: `_run_guards` wrapped in `_reload_lock`

**File:** `ez/agent/sandbox.py`

Without the lock, a concurrent thread iterating `Factor._registry` during the ~3-8ms guard window could observe the probe class with `__module__` starting with `_guard_probe.*`. Probability low under single-worker dev mode but real under multi-thread access.

Wrapped the entire `load_user_class → suite.run → drop_probe` sequence in `with _reload_lock:`. Verified `_run_guards` internals never re-acquire `_reload_lock` (would deadlock — `Lock` is not `RLock`):

- `load_user_class`: imports user file via `importlib.util`, no sandbox calls
- `GuardSuite.run`: calls user code via `invoke_user_code`, no sandbox calls
- `drop_probe_module`: pops dict + del `sys.modules`, no sandbox calls

Hooks calling `_run_guards` followed by `_reload_*_code` is sequential acquire/release/acquire, not nested.

### S5 (partial): Documented date-dependent non-determinism limitation

**File:** `ez/testing/guards/lookahead.py`

The 5-run preflight runs at single `CUTOFF_IDX=150`. Date-dependent non-deterministic strategies (e.g. `if target_date.day == 15: random.random()`) pass preflight but trigger probe BLOCK at other dates. **Doc-only note added with the workaround** (per-probe preflight, +35 invokes / +150ms) and rationale for deferring.

### S7: Suppress collision warnings during guard probe imports

**Files:** `ez/factor/base.py`, `ez/portfolio/cross_factor.py`, `ez/portfolio/portfolio_strategy.py`

Each `__init_subclass__` collision-detection branch now skips the warning when `cls.__module__.startswith("_guard_probe.")`:

```python
if existing is not None and existing is not cls:
    if not cls.__module__.startswith("_guard_probe."):
        logger.warning("Factor name collision: ...")
```

Verified zero "collision" warnings in pytest output across the full test suite.

### Specific concerns for codex on Part 1

**P1 #1 reverse-iteration fix correctness under edge cases** (`drop_probe_module` in `ez/testing/guards/suite.py:174-200`):

The fix uses `for cls in reversed(list(full_keyed.values()))` to walk the dict in reverse insertion order. Edge case I'm uncertain about:

```python
missing = {cls.__name__ for cls in full_keyed.values() if cls.__name__ not in name_keyed}
if missing:
    for cls in reversed(list(full_keyed.values())):
        if cls.__name__ in missing:
            name_keyed[cls.__name__] = cls
            missing.discard(cls.__name__)
            if not missing:
                break
```

1. If `full_keyed` has multiple same-named classes, this picks the last-inserted one. But what if the last-inserted is itself the probe (because the probe was inserted RIGHT before the drop)? After drop pops the probe, the reverse walk should NOT see the probe. **Verify**: the `drop_probe_module` step 1 pops everything where `v.__module__ == module_name` (probe module), so the probe is gone from `full_keyed` before step 2 starts. ✓

2. Concurrency: `_run_guards` is now wrapped in `_reload_lock`. But `drop_probe_module` itself takes no lock. If `drop_probe` is called outside `_run_guards` (e.g. test), and another thread is iterating `full_keyed`, the reverse walk could observe a partially-mutated dict. Is this a real concern? Currently `drop_probe_module` is only called from `_run_guards` (which holds the lock) and from tests (single-threaded), so I think it's fine but worth confirming.

**S4 lock reentrance verification**:

I claim `_run_guards` internals never re-acquire `_reload_lock`. Please trace:
- `load_user_class` → `importlib.util.spec_from_file_location` → user module's top-level code → user imports of `ez.factor.base` etc → `__init_subclass__` (NOT wrapped in lock) → registry mutation. **Question**: can user code do anything that calls back into `sandbox.py` and tries to acquire `_reload_lock`? AST security check forbids `os/sys/subprocess/importlib` etc, but allows `from ez.factor.base import Factor`. Could a malicious factor file `__init__` somehow trigger sandbox functions?

**S7 false-negative risk**:

If a real same-named collision happens AND one of the colliding modules is named `_guard_probe.*` for some unrelated reason (e.g., user names their module that), the warning is suppressed. We rely on `_guard_probe.` being a "private" prefix that no one else uses. Acceptable?

### Core types (`ez/research/context.py`)

```python
@dataclass
class StepRecord:
    step_name: str
    started_at: datetime
    finished_at: datetime
    duration_ms: float
    status: str  # "success" | "failed"
    error: str | None = None
    written_keys: tuple[str, ...] = ()


@dataclass
class PipelineContext:
    config: dict[str, Any] = field(default_factory=dict)       # static config (read-only by convention)
    artifacts: dict[str, Any] = field(default_factory=dict)    # mutable, step output bus
    history: list[StepRecord] = field(default_factory=list)    # append-only audit log

    def get(self, key, default=None) -> Any: ...
    def require(self, key) -> Any:  # raises KeyError with "available keys" hint
```

### Pipeline (`ez/research/pipeline.py`)

```python
class ResearchPipeline:
    def __init__(self, steps: Iterable[ResearchStep]):
        self.steps = list(steps)
        if not self.steps:
            raise ValueError("ResearchPipeline requires at least one step")

    def run(self, context: PipelineContext | None = None) -> PipelineContext:
        ctx = context if context is not None else PipelineContext()
        for step in self.steps:
            t0 = time.perf_counter()
            started = datetime.now()
            try:
                pre_keys = set(ctx.artifacts.keys())
                ctx = step.run(ctx)
                if ctx is None:
                    raise StepError(step.name, TypeError("step.run returned None"))
                written = tuple(sorted(set(ctx.artifacts.keys()) - pre_keys))
                ctx.history.append(StepRecord(..., status="success", written_keys=written))
            except StepError:
                raise
            except Exception as e:
                ctx.history.append(StepRecord(..., status="failed", error=...))
                raise StepError(step.name, e) from e
        return ctx
```

Each step's input/output is recorded as a `StepRecord` for audit. On failure: pipeline stops immediately, context is partially populated, `StepError` includes the original exception.

### RunStrategiesStep

```python
class RunStrategiesStep(ResearchStep):
    name = "run_strategies"
    writes = ("returns", "metrics", "equity_curves")

    def __init__(self, strategies: dict[str, Any], initial_capital: float = 1_000_000.0):
        # strategies dict key is BOTH the result column label AND the
        # universe_data symbol key. No symbol_map yet.
        if not strategies:
            raise ValueError("RunStrategiesStep requires at least one strategy")
        self.strategies = dict(strategies)
        self.initial_capital = float(initial_capital)

    def _run_one(self, df, strategy):
        from ez.backtest.engine import VectorizedBacktestEngine  # lazy
        engine = VectorizedBacktestEngine()
        result = engine.run(df, strategy, self.initial_capital)
        return result.daily_returns, dict(result.metrics), result.equity_curve

    def run(self, context):
        ud = context.require("universe_data")
        returns_dict, metrics_dict, equity_dict, skipped = {}, {}, {}, []
        for label, strategy in self.strategies.items():
            if label not in ud:
                skipped.append((label, f"symbol '{label}' not in universe_data"))
                continue
            try:
                rets, metrics, equity = self._run_one(ud[label], strategy)
            except Exception as e:
                skipped.append((label, f"{type(e).__name__}: {e}"))
                continue
            returns_dict[label] = rets
            metrics_dict[label] = metrics
            equity_dict[label] = equity
        if not returns_dict:
            raise RuntimeError(...)  # all failed
        context.artifacts["returns"] = pd.DataFrame(returns_dict)
        context.artifacts["metrics"] = metrics_dict
        context.artifacts["equity_curves"] = equity_dict
        if skipped:
            context.artifacts["run_strategies_skipped"] = skipped
        return context
```

### ReportStep

```python
def default_template(context: PipelineContext) -> str:
    """Pure-Python markdown rendering. No jinja2 dependency."""
    lines = []
    title = context.config.get("title", "Research Report")
    lines.append(f"# {title}")
    lines.append("")
    lines.append(f"_Generated: {datetime.now().strftime('%Y-%m-%d %H:%M:%S')}_")
    # ... config section, metrics table, returns sample, audit log, warnings ...
    return "\n".join(lines)


class ReportStep(ResearchStep):
    def __init__(self, template_fn=None, output_path=None):
        self.template_fn = template_fn or default_template
        self.output_path = Path(output_path) if output_path else None

    def run(self, context):
        report_md = self.template_fn(context)
        context.artifacts["report"] = report_md
        if self.output_path:
            self.output_path.parent.mkdir(parents=True, exist_ok=True)
            self.output_path.write_text(report_md, encoding="utf-8")
            context.artifacts["report_path"] = str(self.output_path)
        return context
```

### Test isolation rationale

All test Strategy/Factor classes are **duck-typed** (no ABC inheritance):

```python
class _NoopFactor:
    name = "noop"
    warmup_period = 0
    def compute(self, data):
        out = data.copy()
        out[self.name] = 0.0
        return out


class _BuyHoldStrategy:  # NOT inheriting Strategy
    def required_factors(self):
        return [_NoopFactor()]
    def generate_signals(self, data):
        return pd.Series([1.0] * len(data), index=data.index)
```

**Reason**: if `_RaisingStrategy` inherited from `Strategy`, `__init_subclass__` would auto-register it into `Strategy._registry`. Then `tests/test_strategy/test_strategy_contract.py` discovers all registered strategies at module-collection time and tries to run them (calling `generate_signals(sample_df)`). The "raising" strategy intentionally raises → contract test fails.

The `VectorizedBacktestEngine.run()` accepts duck-typed objects (it only calls `strategy.required_factors()` and `strategy.generate_signals(df)`, no `isinstance` check), so the duck-typed test classes work correctly with the real engine.

**Trade-off**: the duck-typed test classes don't catch missing methods at definition time — only at run time. For test code (which is supposed to be obviously simple), I think this is fine. The alternative (ABC + module-teardown unregister) was tried first but failed because `test_strategy_contract.py` collects fixtures at MODULE LOAD time, before any teardown can run.

## Specific concerns for codex on Part 2

### Architecture-level

**A1. ez.research vs ez.agent.runner boundary**

ez.agent.runner:
- Single experiment: `RunSpec → RunResult` (backtest + WFO + significance)
- Has its own gate framework (ResearchGate)
- Has its own persistence (ExperimentStore)

ez.research:
- Multi-step orchestration: `pipeline.run(context) → context with artifacts`
- No gate, no persistence (yet)
- Steps can call ez.agent.runner internally (currently RunStrategiesStep uses VectorizedBacktestEngine directly, not Runner)

**Question**: Is the boundary right? Should `RunStrategiesStep` go through `ez.agent.Runner` (to get WFO + significance + gate for free), or stay with `engine.run()` (lower-level, less coupling)? The current choice was "engine.run is simpler and the step doesn't need WFO" but I'm not sure that's the long-term right call. Future `RunStrategiesStep` might want to declare `with_wfo: bool` and route through Runner.

**A2. Lazy import discipline**

`DataLoadStep._fetch_one` and `RunStrategiesStep._run_one` lazy-import `ez.api.deps` and `ez.backtest.engine` at call time, not at module-load time. Goal: keep `ez.research` importable in environments where `ez.api` (FastAPI) or `ez.backtest` (numpy/pandas-heavy) aren't fully ready.

**Question**: Is lazy import worth the trade-off (delayed ImportError until step runs vs. clean module loading)? In practice the test environment imports `ez.api.deps` for other tests anyway, so lazy import doesn't actually help in tests. It only helps if someone tries to import `ez.research.steps.data_load` standalone in a minimal deployment — which is plausible for V2.20.x scripted use but doesn't exist today.

**A3. PipelineContext mutability and ownership**

```python
config: dict      # "read-only by convention" — not enforced
artifacts: dict   # mutable, free-for-all
history: list     # append-only by convention
```

No type schema for `artifacts` keys (just `str` → `Any`). Two steps can write the same key — the second wins, and `written_keys` for the second step won't include the key (because `pre_keys = set(ctx.artifacts.keys())` already had it). **Question**: should this be a hard error or warning? Currently silent overwrite.

No way to declare "this step REQUIRES key X" at construction time — only at runtime via `context.require(key)`. Could be auto-validated by inspecting `step.writes` declarations across the pipeline before run. **Question**: worth doing in V2.20.0 or defer?

**A4. Pipeline error semantics**

On any step failure: pipeline raises `StepError`, no retry, no continuation, partial context returned in the exception's traceback (caller can `try/except StepError` and inspect `ctx.history`). **Question**: is no-retry too aggressive? Some steps (DataLoadStep network fetch) might benefit from automatic retry. Currently the user has to wrap the step themselves.

**A5. Memory model**

`artifacts['universe_data']` is `dict[symbol → DataFrame]`. For a 10-year × 100-symbol study that's ~1GB in memory. The pipeline holds the entire context across all steps. **Question**: should there be a way to drop intermediate artifacts after they're consumed (e.g., `RunStrategiesStep.run` could delete `universe_data` after extracting what it needs)? Currently no eviction.

### API-level

**B1. RunStrategiesStep symbol = label assumption**

**B2. DataLoadStep `_resolve` precedence**

```python
def _resolve(self, context):
    symbols = self.symbols if self.symbols is not None else context.config.get("symbols")
    start = self.start_date or context.config.get("start_date")
    end = self.end_date or context.config.get("end_date")
    market = self.market if self.market != "cn_stock" else context.config.get("market", "cn_stock")
    period = self.period if self.period != "daily" else context.config.get("period", "daily")
```

The `market != "cn_stock"` and `period != "daily"` checks mean the user CANNOT explicitly pass `market="cn_stock"` and have it NOT fall back to context config. If someone wants `market="cn_stock"` constructor + `market="us_stock"` config, the constructor's value is silently overridden because `self.market == "cn_stock"` matches the default sentinel. **Question**: should I use `None` as the default sentinel instead? (`market: str | None = None` and `... if self.market is not None else config.get(...)`.)

**B3. ReportStep template signature**

```python
def template_fn(context: PipelineContext) -> str: ...
```

Custom template gets the entire context (config + artifacts + history). **Question**: is exposing the full context too much? Could pass a curated dict instead. But that requires knowing what keys to expose, which couples the template fn to the pipeline structure.

### Test-quality

**C1. test_pipeline_chains_data_load_and_run_strategies + test_full_pipeline_*** tests use `monkeypatch.setattr(DataLoadStep, "_fetch_one", fake_fetch)`. These tests are essentially monkey-patching at the class level — every instance shares the patch. **Question**: is this test pattern robust enough? An alternative would be a `--no-network` pytest marker or a `data_chain` fixture.

**C2. test_full_pipeline_partial_failure_records_warning** verifies the report contains warnings. But it does NOT verify the report's MARKDOWN STRUCTURE is well-formed (e.g., that the table syntax is correct, that there are no broken references). The assertions are substring checks. **Question**: should there be a markdown lint test? Or is "report contains 'planned crash'" enough?

**C3. ReportStep default_template** has hard-coded sections in a fixed order: title, config, metrics table, returns sample, audit log, warnings. There are no tests for "section ordering is consistent" or "markdown links are valid" or "table column count matches header". The 11 tests verify each section RENDERS but not that the WHOLE document is well-formed. **Question**: acceptable for MVP?

### Things I haven't tested

**D1. Pipeline with > 5 steps**: not tested. The framework supports any number, but I haven't stress-tested. Possibly the `for step in self.steps:` loop has an issue I didn't catch.

**D2. Re-running the same pipeline twice**: not tested. Each `pipeline.run(context)` mutates the context. If you call `pipeline.run(ctx)` twice with the same ctx, the second run will see populated artifacts from the first. Is this intentional? Currently undocumented.

**D3. Steps that take other steps as constructor args** (composite steps): not tested. The framework supports it (steps are just objects), but no example exists.

**D4. Real `validation/phase_o_nested_oos.py` migration**: I have NOT yet rewritten an actual phase script using `ez.research`. The MVP claim "you can replace 200 lines with 20" is unverified.

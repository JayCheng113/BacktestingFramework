## Root Cause Analysis: Why This Phase Exists

Phase 4 wired `ml_alpha` into the sandbox (save/validate/hot-reload). But there's a **startup-scan gap**: when the server restarts, `loader.py` scans `cross_factors/` and `portfolio_strategies/` to import user files and register their classes in the CrossSectionalFactor registry. It does **NOT** scan `ml_alphas/`. Result: after a restart, user-written MLAlpha subclasses vanish from the registry until someone manually triggers `/api/code/refresh` or re-saves the file.

This is the same pattern as `cross_factors/` — the fix is a 3-line function in `loader.py` + a call at startup.

The diagnostics endpoint is new functionality: there's no existing endpoint that runs `MLDiagnostics` and returns the result. Phase 2 built the Python API; Phase 5 exposes it via REST.

- [ ] **Step 5.2.2: Implement endpoint**

```python
@router.post("/ml-alpha/diagnostics")
def ml_alpha_diagnostics(req: MLDiagnosticsRequest):
    """Run MLDiagnostics on a user-defined MLAlpha and return the
    overfitting assessment.

    V2.13.1 Phase 5: exposes the Phase 2 Python API (MLDiagnostics)
    as a REST endpoint for Phase 6 frontend consumption.

    Resolution: CrossSectionalFactor.resolve_class(ml_alpha_name)
    → must be an MLAlpha subclass (not just any CrossSectionalFactor).

    Error handling:
    - Class not found → 404
    - Not an MLAlpha subclass → 422
    - Instantiation fails (whitelist/n_jobs) → 422
    - Data fetch fails → 502
    - Diagnostics runs but verdict is "insufficient_data" → 200 (valid result)
    """
    from ez.portfolio.cross_factor import CrossSectionalFactor
    from ez.portfolio.ml_alpha import MLAlpha
    from ez.portfolio.ml_diagnostics import MLDiagnostics, DiagnosticsConfig

    # 1. Resolve class
    try:
        cls = CrossSectionalFactor.resolve_class(req.ml_alpha_name)
    except (KeyError, ValueError) as e:
        raise HTTPException(status_code=404, detail=f"MLAlpha '{req.ml_alpha_name}' not found: {e}")

    if not issubclass(cls, MLAlpha):
        raise HTTPException(
            status_code=422,
            detail=f"'{req.ml_alpha_name}' is a {cls.__name__} "
                   f"(CrossSectionalFactor), not an MLAlpha subclass. "
                   f"Use /evaluate-factors for non-ML factors.",
        )

    # 2. Instantiate (may raise UnsupportedEstimatorError)
    try:
        alpha = cls()
    except Exception as e:
        raise HTTPException(
            status_code=422,
            detail=f"Failed to instantiate {req.ml_alpha_name}: "
                   f"{type(e).__name__}: {e}",
        )

    # 3. Fetch data (reuse existing _fetch_data helper)
    end = req.end_date or date.today()
    start = req.start_date or (end - timedelta(days=730))  # 2 years
    try:
        universe_data, calendar = _fetch_data(
            req.symbols, req.market, start, end,
            lookback_override=alpha.warmup_period + 50,
        )
    except Exception as e:
        raise HTTPException(
            status_code=502,
            detail=f"Data fetch failed: {type(e).__name__}: {e}",
        )

    # 4. Build config + run
    config = DiagnosticsConfig(
        severe_overfit_threshold=req.severe_overfit_threshold,
        mild_overfit_threshold=req.mild_overfit_threshold,
        high_turnover_threshold=req.high_turnover_threshold,
        top_n_for_turnover=req.top_n_for_turnover,
        forward_horizon=req.forward_horizon,
    )
    diag = MLDiagnostics(alpha, config=config)
    result = diag.run(universe_data, calendar, start, end, req.eval_freq)

    # 5. Return JSON-serializable dict
    return result.to_dict()
```

**Note on `_fetch_data`:** The existing helper in `portfolio.py` fetches universe data with a configurable lookback. If `_fetch_data` doesn't accept a `lookback_override` parameter, we'll need to pass `alpha.warmup_period + 50` through whatever mechanism it supports. Read the actual `_fetch_data` signature during implementation and adapt.

## Error Handling Matrix

| Scenario | HTTP Status | Response |
|---|---|---|
| `ml_alpha_name` not in registry | 404 | `{"detail": "MLAlpha 'X' not found: KeyError(...)"}` |
| Name resolves to non-MLAlpha (e.g., MomentumRank) | 422 | `{"detail": "'MomentumRank' is a MomentumRank (...), not an MLAlpha subclass"}` |
| MLAlpha instantiation fails (whitelist, n_jobs) | 422 | `{"detail": "Failed to instantiate X: UnsupportedEstimatorError: ..."}` |
| Data fetch fails (no provider, network) | 502 | `{"detail": "Data fetch failed: ProviderError: ..."}` |
| Symbols list empty | 422 | Pydantic validation error |
| Diagnostics succeeds but verdict = "insufficient_data" | 200 | Valid JSON with `"verdict": "insufficient_data"` + `"warnings": [...]` |
| Diagnostics succeeds normally | 200 | Full DiagnosticsResult.to_dict() JSON |

## Anti-patterns to avoid

1. **Do NOT add a parallel `/ml-alpha/evaluate` endpoint.** MLAlpha is just another CrossSectionalFactor — the existing `/evaluate-factors` handles it. The diagnostics endpoint is NEW functionality (overfitting detection), not a duplicate of factor evaluation.

2. **Do NOT modify `_get_factor_map()`.** MLAlpha subclasses auto-register in CrossSectionalFactor._registry via `__init_subclass__`. The factor map already iterates that registry. The only missing piece was the startup scan (`load_ml_alphas()`).

3. **Do NOT import MLAlpha/MLDiagnostics at module level in portfolio.py.** Use lazy imports inside the endpoint function to avoid circular dependencies and to keep the import optional (sklearn may not be installed).

4. **Do NOT hardcode test symbols or expect specific data.** Tests that call real data providers must handle 502/500 gracefully (skip in CI). Use the `pytest.skip` pattern, not `pytest.xfail`.

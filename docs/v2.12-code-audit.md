# V2.12 Code Logic Audit Document

## Audit Scope

逐文件、逐函数审查 V2.12 新增/修改的所有代码，检查：
1. **正确性** — 数学公式、逻辑分支、边界条件
2. **健壮性** — NaN/Inf 守卫、空数据、异常路径
3. **效率** — 热路径复杂度、不必要的重复计算
4. **交叉 Bug** — 与现有功能的交互是否安全

## A. `ez/portfolio/optimizer.py`

### A1. `ledoit_wolf_shrinkage()`

| 检查项 | 行号 | 检查内容 |
|--------|------|---------|
| A1.1 | 36-37 | T<2 fallback: 返回 0.04*I (20% vol)。是否合理？如果全部 returns 为 0，T>=2 时 S=0, trace_S=0, mu=0, F=0, sigma=0+ridge。与 T<2 的 0.04 不一致 |
| A1.2 | 39 | `np.cov(returns, rowvar=False, ddof=1)`: 当 T==2 时 ddof=1 产生一个自由度，S 方差可能很大。是否需要 min(T) 守卫？ |
| A1.3 | 44 | `trace_S2 = np.sum(S ** 2)`: 这是 Frobenius 范数平方，用于 OAS 公式。正确 |
| A1.4 | 49-50 | OAS 公式: `rho_num` 和 `rho_den` 是否匹配 Chen 2010？需验证符号和系数 |
| A1.5 | 51 | `abs(rho_den) > 1e-20`: 当 `trace_S2 == trace_S**2 / N` 时 den=0。此时 rho=1.0 (全收缩)，合理 |
| A1.6 | 39-41 | N==1 特判: `S = np.cov(...)` 当 N=1 时返回标量不是矩阵。`S.reshape(1,1)` 是否安全？ |

### A6. `RiskParityOptimizer._optimize()`

| 检查项 | 行号 | 检查内容 |
|--------|------|---------|
| A6.1 | 264-265 | `port_var < 1e-20: return 0.0`。当 w 极小时 port_var 趋近 0，rc 的梯度不连续。SLSQP 可能在此点震荡 |
| A6.2 | 268 | `rc = w * marginal / port_var`: w 和 marginal 都是向量，port_var 是标量。rc 是风险贡献比例。sum(rc) 应该 = 1.0（验证：sum(w * Σw / w'Σw) = w'Σw / w'Σw = 1）。正确 |
| A6.3 | 277 | bounds `[(1e-6, max_w)]`: 下界 1e-6 而非 0，避免 0 权重导致 rc 梯度奇异。但如果 n > 1/max_w，sum(n * 1e-6) > 0 但远小于 1，不影响 |
| A6.4 | 286 | fallback `return w0`: w0 是 inverse-vol，可能不满足 max_w 约束。但后续 `_apply_constraints` 会 clip。安全 |

### B2. `check_turnover()`

| 检查项 | 行号 | 检查内容 |
|--------|------|---------|
| B2.1 | 64-67 | `actual_turnover = sum(abs(new - old))`: 这是双边换手率（买+卖总量），不是单边。如果 A 从 0→50%, B 从 50%→0%: turnover=1.0 (100%)。文档说"单边"但代码算的是双边 |
| B2.2 | 71 | `alpha = max_turnover / actual_turnover`: 如果 actual=0 (new==old)，已被 line 68 guard。安全 |
| B2.3 | 73-76 | 混合后权重和: `alpha * sum(new) + (1-alpha) * sum(old)`。如果 new 和 old 都 sum=1，混合也 sum=1。但如果 new sum < 1（优化器留了现金），混合会拉向 old 的权重。可能导致 sum > 1 |

## C. `ez/portfolio/engine.py` (V2.12 modifications)

### C1. Daily drawdown + emergency sell

| 检查项 | 行号 | 检查内容 |
|--------|------|---------|
| C1.1 | 172 | `dd_scale = 1.0`: 在 risk_manager is None 时保持 1.0。line 242 `if dd_scale < 1.0` 不会触发。安全 |
| C1.2 | 181 | `dd_event and "减仓" in dd_event`: 只在首次 breach 时紧急减仓。但如果用户自定义 event 字符串不含"减仓"（比如英文），判断失败。当前 RiskManager 硬编码中文，耦合 |
| C1.3 | 184 | `_lot_round(holdings[sym] * dd_scale, lot_size)`: dd_scale=0.5, holdings=100: target=_lot_round(50, 100)=0。持仓 100 → 0，全部清仓。如果 dd_scale=0.5, holdings=300: target=_lot_round(150,100)=100。只保留 100 股 |
| C1.4 | 186 | `delta >= 0`: 如果 target >= holdings (不需要卖)，跳过。但 target = holdings*scale 且 scale<1，所以 target < holdings，delta < 0。除非 lot_round 向上取整到 holdings。不会：_lot_round 向下取整 |
| C1.5 | 197-199 | `cash += sell_amount - comm - stamp`: commission 从 cash 扣除。如果 sell_amount 很小但 min_commission=5，comm > sell_amount？`_compute_commission(amount, rate, minimum)` 返回 `max(amount*rate, minimum)`。如果 amount*rate < 5, comm=5, stamp ≈ 0。cash += sell_amount - 5。如果 sell_amount < 5, cash 减少。这可能导致 cash < 0（小额紧急减仓的 cost > 收入） |

### C2. Rebalance path

| 检查项 | 行号 | 检查内容 |
|--------|------|---------|
| C2.1 | 228-230 | `optimizer.set_context(day, sliced_tradeable)`: sliced_tradeable 是截至 day-1 的数据。optimizer._estimate_covariance 用 `< current_date` 过滤。如果 set_context 传的是 day（而非 day-1），过滤 `< day` 正确排除当天数据。但 sliced_tradeable 已经做了 anti-lookahead slice，所以 optimizer 看到的数据范围是 [day-lookback, day-1]，再用 `< day` 过滤是冗余的（但无害） |
| C2.2 | 236 | `check_turnover(raw_weights, prev_weights)`: prev_weights 来自上次 rebalance 后的实际权重（line 302-306）。但如果紧急减仓发生在两次 rebalance 之间，prev_weights 不反映紧急减仓后的实际持仓。换手率计算基于 "上次 rebalance 权重" 而非 "当前实际权重"，可能低估实际换手 |
| C2.3 | 242-243 | `dd_scale < 1.0: raw_weights *= dd_scale`: 这在已 breached 但未恢复时每次 rebalance 都 scale。累积效果：breached 期间每次 rebalance 权重都是 scale * 策略权重。不是指数衰减（因为权重是基于新策略输出乘固定 scale），是合理的 |

## D. `ez/portfolio/attribution.py`

### D1. `_period_return()`

| 检查项 | 行号 | 检查内容 |
|--------|------|---------|
| D1.1 | 43-44 | `df.index.date >= start`: start 是 rebalance_dates[i]。如果第一个 rebalance_date 之前有数据，p0 取的是 start 当天的价格，不是 start-1。这和 engine 的 "entry at rebalance date" 是否一致？Engine 在 rebalance 日执行交易，所以 start 日开始持仓。用 start 日价格作为 p0 是正确的 |

### D2. `compute_attribution()`

| 检查项 | 行号 | 检查内容 |
|--------|------|---------|
| D2.1 | 96 | `min(len(rebalance_dates) - 1, len(weights_history))`: weights_history 可能比 rebalance_dates 短（如果最后一次 rebalance 没有成功生成权重）。用 min 防越界。正确 |
| D2.2 | 99 | `w_p = weights_history[i]`: 这是 rebalance_dates[i] 时刻的权重。但 weights_history 在 engine 中是每个 rebalance 日 append（line ~302），索引对齐正确 |
| D2.3 | 103-106 | 等权基准: `active` 包含所有在 [t_start, t_end] 有数据的 universe symbols，不仅是组合持有的。这意味着基准可能包含组合未持有的股票。这是正确的 Brinson 行为 — 基准是全 universe 等权 |
| D2.4 | 118 | `industry_map.get(s, "_other")`: 没有行业映射的股票归入 "_other"。如果 industry_map 为空（没有 FundamentalStore），所有股票都在 "_other"，选股效应全部归入 "_other"。归因结果不太有意义但不会出错 |
| D2.5 | 125-126 | `r_p_j = _weighted_return(syms, w_p, ...) if w_p_j > 0 else 0`: 如果组合在某行业权重为 0，r_p_j=0。对应的 selection effect = w_b_j * (0 - r_b_j)。这意味着组合未持有的行业产生负的选股效应。这在 Brinson 框架下是正确的 — "选择不持有" 也是一种选股决策 |

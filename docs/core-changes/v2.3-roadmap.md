# V2.3+ Roadmap — Based on Codex Audit Recommendations

## Principle: Correctness Ceiling Before Performance Expansion

> "先做正确性封顶，再做性能扩张" — Codex

## Phase A: Correctness Hardening (V2.3)

### A1. Accounting Invariant Test (per-bar reconciliation)
每一 bar 验证：`cash + position_value == equity_curve[i]`
且全局：`initial_capital + sum(realized_pnl) - sum(fees) + unrealized_pnl == final_equity`

### A2. Python/C++ Dual-Path Comparison Framework
同输入同参数，C++ on/off 两条路径跑完，逐列数值对比。纳入 CI。
覆盖：ts_ops 5 函数 × (clean data + NaN data + edge cases)

### A3. Parameter Semantics Unification
文档化所有函数的参数约束：
- window/span/periods: must be > 0 (enforced in Python wrapper + C++)
- NaN handling: skip in accumulation, NaN output when valid count < min_periods
- inf handling: IEEE 754 natural (pct_change div-by-zero → inf)
- Default values: from config.yaml, API overrides optional

## Phase B: Targeted Performance (V2.4)

### B1. rolling_std Welford Online Algorithm
当前 O(n*window)，优化为 O(n)。已标注 TODO，是最低挂的果。

### B2. engine._simulate C++ 下沉 (评估)
热循环移入 C++。需要评估：
- 收益：5000 bars 82ms → 目标 <10ms
- 代价：nanobind 回调 Strategy.generate_signals 的开销
- 风险：trade recording 逻辑复杂，C++ 实现易出 bug
结论：仅在日线吞吐不足时做。日线场景当前 60K bars/sec 已足够。

### B3. Monte Carlo/Bootstrap C++ (评估)
significance.py 的 permutation loop 适合 C++。但当前只跑 1000 次，耗时 ~10ms。
结论：仅在需要 10000+ permutations 时做。

## Phase C: Feature Expansion (V2.5+)

### C1. Multi-Asset Backtesting
- 多标的同时回测
- 截面策略支持
- PortfolioState 完善

### C2. 更多数据源
- Tiingo, Polygon, Binance (crypto)
- 按需实现

## Not Planned (YAGNI)
- Event-driven engine: 无 tick 数据，不做
- CRTP/Eigen 重度 C++: 当前 nanobind 薄封装够用
- SIMD 显式优化: 编译器自动向量化已足够

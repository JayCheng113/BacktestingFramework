# V2.3+ Roadmap

## Principle

> "先做正确性封顶，再做 Agent 闭环" — Codex + 内部共识
>
> "核心不是少做功能，而是给功能设生命周期和预算"

治理规则见 `docs/architecture/governance.md`。

---

## V2.3 — Correctness Hardening (正确性封顶)

**目标**：把现有研究链路做成"可证明正确"。

**模块**：ez/backtest, ez/core/ts_ops.py, ez/core/_cpp/ts_ops.cpp, tests/

### 交付

- [ ] A1. 会计不变量测试：逐 bar `abs(cash + position_value - equity[i]) <= 0.01`
- [ ] A2. C++/Python 双路径对拍框架：同输入 _USE_CPP=True/False，结果逐列比对
- [ ] A3. rolling_std O(n) Welford 优化
- [ ] A4. 架构门禁测试 (tests/test_architecture/)

### 清理

- [ ] 统一参数验证入口到 ts_ops.py（Python 和 C++ 双层校验已有，确认无遗漏）
- [ ] 确认 eps 容差在所有不变量测试中一致 (0.01)

### Exit Gate

- [ ] 全量测试通过（含新增 invariant + 对拍）
- [ ] C++/Python 对拍差异 < eps (0.01)
- [ ] rolling_std C++ 性能优于 Python
- [ ] 性能基线不回退（benchmark.py 结果 >= V2.2）
- [ ] 0 新增 Critical/Important bug

## V2.4 — Agent Loop (Agent 闭环)

**目标**：最小可用 Agent research loop。

**新模块**：ez/agent/ (experimental)

### 交付

- [ ] B1. RunSpec — 标准化实验输入 (`ez/agent/run_spec.py`)
- [ ] B2. Runner — 组合执行 backtest + WFO + significance (`ez/agent/runner.py`)
- [ ] B3. Gate — 自动晋级规则 + 原因码 (`ez/agent/gates.py`)
- [ ] B4. Report — 结构化输出 (`ez/agent/report.py`)
- [ ] B5. 实验持久化 — DuckDB runs/run_artifacts/gate_verdicts 表 (`ez/data/store.py`)
- [ ] B6. API — `/experiments` 端点 (`ez/api/routes/experiments.py`)
- [ ] B7. Web — 实验管理页面（列表+对比+Gate结果）

### 清理

- [ ] 下线绕过 Runner 的 ad-hoc 回测入口（API 直接调引擎仍保留，但建议用 Runner）

### Exit Gate

- [ ] 实验全链路：submit → execute → gate → query → compare
- [ ] Gate 判定与 significance/WFO 结果一致
- [ ] 审计字段完整（run_id, code_commit, timestamp）
- [ ] 幂等性：同一 RunSpec 重复提交不产生重复实验

# V2.0 Core Change Proposal: Matcher Extraction + ts_ops

## Motivation

V2 requires C++ acceleration of hot paths. Current `engine._simulate()` mixes orchestration
with fill/commission logic — impossible to replace selectively. Factor code calls pandas
directly — no seam for C++ replacement.

## Changes

### 1. New package: `ez/core/`

```
ez/core/
    __init__.py
    matcher.py      # FillResult + Matcher ABC + SimpleMatcher
    ts_ops.py       # rolling_mean, rolling_std, ewm_mean, diff, pct_change (pandas wrappers)
```

**Why `ez/core/` not `ez/backtest/matcher.py`?**
ts_ops serves both factor and backtest layers. A shared `core` package avoids circular deps
and becomes the natural home for C++ bindings in V2.1.

### 2. Modified core files

| File | Change | Risk |
|------|--------|------|
| `ez/backtest/engine.py` | `_simulate()` delegates to `Matcher.fill_buy/fill_sell` | Medium — logic extraction, behavior preserved |
| `ez/factor/builtin/technical.py` | Use `ts_ops.*` instead of `df.rolling()` / `df.ewm()` | Low — thin wrappers, identical output |

### 3. Interface design

```python
@dataclass
class FillResult:
    shares: float       # shares filled
    fill_price: float   # execution price
    commission: float   # commission charged
    net_amount: float   # cash delta (negative=buy, positive=sell)

class Matcher(ABC):
    def fill_buy(self, price: float, amount: float) -> FillResult: ...
    def fill_sell(self, price: float, shares: float) -> FillResult: ...
```

Engine receives `Matcher` via constructor (default=`SimpleMatcher`).
V2.1 adds `SlippageMatcher`, V2.2 adds `EventDrivenMatcher` — all same interface.
